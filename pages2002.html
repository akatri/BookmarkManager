/nWhen should you use an abstract class, when an/ninterface, when both? Interfaces and abstract classes seem superficially to provide almost the same/ncapability. How do you decide which to use? /n/n/n/n/n/n/n/n

When To Use Interfaces

/nAn interface allows somebody to start from scratch to implement your/ninterface or implement your interface in some other code whose original or primary purpose was/nquite different from your interface. To them, your/ninterface is only incidental, something that have to add/non to the their code to be able to use your package. The disadvantage is every method/nin the interface must be public. You might not want to/nexpose everything./n

When To Use Abstract classes

/nAn abstract class, in contrast, provides more structure. It usually/ndefines some default implementations and provides some tools useful for a full/nimplementation. The catch is, code using it must use your class as/nthe base. That may be highly inconvenient if the other programmers wanting to use/nyour package have already developed their own class hierarchy independently. In Java,/na class can inherit from only one base class./n

When to Use Both

/nYou can offer the best of both worlds, an/ninterface and an abstract/nclass. Implementors can ignore your abstract class if/nthey choose. The only drawback of doing that is calling methods via their/ninterface name is slightly slower than calling them via/ntheir abstract class name./n

multiple inheritanceA class may implement several interfaces.A class may extend only one abstract/nclass.
default implementationAn interface cannot provide any code at all,/nmuch less default code.An abstract class can provide complete code,/ndefault code and/or just stubs that have to be overridden.
constantsStatic final constants only, can use them without qualification in classes/nthat implement the interface. On the other paw,/nthese unqualified names pollute the namespace. You can use them and it is not/nobvious where they are coming from since the qualification is optional.Both instance and static constants are possible. Both static and instance/nintialiser code are also possible to compute the constants.
third party convenienceAn interface implementation may be added to/nany existing third party class.A third party class must be rewritten to extend only from the abstract class.
is-a vs -able or can-doInterfaces are often used to describe the peripheral abilities of a class,/nnot its central identity, e.g. An Automobile class/nmight implement the Recyclable interface, which could apply to many otherwise totally/nunrelated objects.An abstract class defines the core identity of/nits descendants. If you defined a Dog abstract/nclass then Dalmatian descendants are Dogs, they are not/nmerely dogable. Implemented interfaces enumerate the general things a class/ncan do, not the things a class is./n

In a Java context, users should typically implement the Runnable/ninterface rather than extending Thread because they’re not really interested in/nproviding some new Thread functionality, they/nnormally just want some code to have the capability of running independently./nThey want to create something that can be run in a thread, not a new kind of/nthread.The similar is-a vs has-a debate comes up when you decide to inherit/nor delegate.

plug-inYou can write a new replacement module for an interface that contains not one stick of code in common with/nthe existing implementations. When you implement the interface, you start from/nscratch without any default implementation. You have to obtain your tools from/nother classes; nothing comes with the interface/nother than a few constants. This gives you freedom to implement a radically/ndifferent internal design.You must use the abstract class as-is for the/ncode base, with all its attendant baggage, good or bad. The abstract class author has imposed structure on you. Depending/non the cleverness of the author of the abstract/nclass, this may be good or bad.
homogeneityIf all the various implementations share is the method signatures, then an/ninterface works best.If the various implementations are all of a kind and share a common status/nand behaviour, usually an abstract class works/nbest. Another issue that’s important is what I call heterogeneous vs. homogeneous. If implementors/subclasses are/nhomogeneous, tend towards an abstract base class./nIf they are heterogeneous, use an interface. (Now/nall I have to do is come up with a good definition of hetero/homo-geneous in/nthis context.) If the various objects are all of-a-kind and share a common/nstate and behavior, then tend towards a common base class. If all they share is/na set of method signatures, then tend towards an interface.
maintenanceIf your client code talks only in terms of an interface, you can easily change the concrete implementation/nbehind it, using a factory/nmethod.Just like an interface, if your client code/ntalks only in terms of an abstract class, you/ncan easily change the concrete implementation behind it, using a factory method.
speedSlow, requires extra indirection to find the corresponding method in the/nactual class. Modern JVMs (Java Virtual Machines)/nare discovering ways to reduce this speed penalty.Fast
tersenessThe constant declarations in an interface are/nall presumed public static final, so you may leave/nthat part out. You can’t call any methods to compute the initial values/nof your constants. You need not declare individual methods of an interface abstract. They are all/npresumed so.You can put shared code into an abstract/nclass, where you cannot into an interface. If/ninterfaces want to share code, you will have to write other bubblegum to/narrange that. You may use methods to compute the initial values of your/nconstants and variables, both instance and static. You must declare all the/nindividual methods of an abstract class/nabstract.
adding functionalityIf you add a new method to an interface, you/nmust track down all implementations of that interface in the universe and provide them with a concrete/nimplementation of that method.If you add a new method to an abstract class,/nyou have the option of providing a default implementation of it. Then all/nexisting code will continue to work without change.
